Interactive prototypes in CC Design are self-contained HTML files that run React entirely in the browser — no build step, no Node.js. Babel transpiles JSX at runtime from CDN-loaded scripts, so you get real component state, multi-screen navigation, and keyboard interactions in a single deliverable file you can open directly.

## How CC Design builds prototypes

When you ask for an interactive prototype, CC Design loads two references and picks an appropriate frame template:

- **`references/react-babel-setup.md`** — pinned CDN versions of React 18.3.1, ReactDOM 18.3.1, and Babel standalone 7.29.0, each with SRI integrity hashes to prevent runtime breakage
- **`references/interactive-prototype.md`** — navigation patterns (tab bar, sidebar, wizard), state management strategies, page transitions, and the full prototype HTML structure

The frame template depends on what you're building. A mobile prototype uses `ios_frame.jsx` or `android_frame.jsx`. A desktop app prototype uses `browser_window.jsx` or `macos_window.jsx`. A standalone web prototype uses no frame.

<Note>
  CC Design always uses pinned versions with integrity hashes — never `react@18` or bare unpkg URLs without hashes. This prevents silent breakage when CDN packages update.
</Note>

## State management

| Complexity | Pattern | When to use |
|---|---|---|
| 2–3 screens | `useState` for active screen index | Tab bar, simple nav |
| 5+ screens | `useReducer` with screen + form state | Wizards, multi-step forms |
| URL sync needed | Hash routing | Back button should work, or sharing specific screen URLs |

Hash routing lets each screen have a bookmarkable URL:

```js
const [screen, setScreen] = React.useState(location.hash.slice(1) || 'home');

React.useEffect(() => {
  const onHash = () => setScreen(location.hash.slice(1) || 'home');
  window.addEventListener('hashchange', onHash);
  return () => window.removeEventListener('hashchange', onHash);
}, []);

// Navigate to a screen:
location.hash = '#settings';
```

## Multi-file scope rules

When a prototype is split across multiple `.jsx` files (e.g. a shared component file plus a main app file), each `<script type="text/babel">` gets its own scope when Babel transpiles it. Components defined in one file are invisible to the others unless explicitly exported to `window`:

```js
// At the end of components.jsx:
Object.assign(window, {
  Sidebar, TabBar, Avatar, Badge,
  // ... all components that other files need
});
```

Also give every style object a unique name — if two files both declare `const styles = { ... }`, the second one overwrites the first:

```js
// Good
const sidebarStyles = { width: '240px', ... };

// Bad — breaks when a second component file does the same
const styles = { width: '240px', ... };
```

## Verification before delivery

Before handing off a prototype, CC Design runs a console error check in the browser and verifies that:

- Navigation between all screens works without errors
- No React key warnings appear in the console
- Screen transitions animate correctly
- Keyboard shortcuts (if any) fire on the correct elements

<Warning>
  If the browser console shows errors, the prototype is not ready to deliver. CC Design re-enters the build step to fix all errors before marking the task complete.
</Warning>
